# TASK-014 — SLIPPER: реже вспомогательные системы (`_process`, таймеры)

## Цель

Найти в `honor-of-aurora/` системы, которые выполняют работу **каждый кадр** или **слишком часто**, и в режиме **«На тапке»** снизить частоту обновления **без изменения** игровой логики критичного пути (бой, ввод, смерть).

## Контекст

- Примеры уже существующих паттернов: `mobile_touch_controls.gd` — пропуск кадров в SLIPPER; `YSortManager` — редкий полный обход.
- Искать: `func _process`, `func _physics_process`, `Timer` с малым интервалом, `call_group` в цикле.

## Требования (MUST)

1. Каждая оптимизация завязана на `PerformancePreset.is_slipper_mode(SaveManager)` (или эквивалент через `SaveManager`).
2. Для каждой изменённой системы:
   - кратко в комментарии: что экономится, **что не должно сломаться**;
   - если используется `Engine.get_process_frames() % N`, документировать `N` и причину.
3. **Не** трогать:
   - обработку ввода так, чтобы появилась задержка управления;
   - физику `CharacterBody2D` / коллизии (если не оговорено отдельно в TASK-011).
4. Приоритет кандидатов: декоративная анимация, погодные вспомогательные узлы, редкие проверки «наведения», обновление оверлеев без состояния.

## Метод работы

1. Grep по `honor-of-aurora/` на `_process(` и `_physics_process(`.
2. Отфильтровать тяжёлые/частые вызовы (по смыслу и по профайлеру при возможности).
3. Внедрить throttle только в SLIPPER.

## Критерии приёмки

- Управление и бой ощущаются как раньше на «На тапке» (в пределах уже принятого FPS из TASK-011).
- Нет спама ошибок в консоль.
- Журнал волны 2 содержит список затронутых файлов.
