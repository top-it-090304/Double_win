# TASK-011 — SLIPPER: более агрессивные параметры движка (волна 2)

## Цель

Дополнительно снизить нагрузку **только в режиме «На тапке»** (`PerformancePreset.Mode.SLIPPER`), изменив параметры движка и глобальной Y-сортировки так, чтобы они были **строже**, чем в первой волне (TASK-002), но без неконтролируемого ломания боёвки.

## Контекст

- Текущая логика: `honor-of-aurora/autoload/performance_preset.gd` → `apply_from_save_manager` → ветка `Mode.SLIPPER` (пример: 30 FPS, 30 Гц физики, `YSortManager.refresh_every_frames` = 8).
- `Engine.max_fps` сейчас дополнительно ограничивается `clampi(max_fps_val, 30, 240)` — **нижняя граница 30** не позволяет официально выставить 20 FPS без изменения этой ветки **для SLIPPER**.
- `YSortManager` (`honor-of-aurora/autoload/YSortManager.gd`): экспорт `refresh_every_frames` с `@export_range(1, 8, 1)` — значения **> 8** потребуют **расширения диапазона** и согласованного `clamp` в `PerformancePreset`.

## Требования (MUST)

1. Все изменения чисел — **только** для `mode == Mode.SLIPPER` в `apply_from_save_manager` (и при необходимости вспомогательные функции/константы рядом с пресетом).
2. Явно задокументировать в коде **комментариями**:
   - целевые значения FPS, `physics_ticks_per_second`, `refresh_every_frames`;
   - **риски**: снижение FPS ниже 30, снижение physics ticks, Y-sort реже 8 кадров (артефакты слоёв, точность коллизий).
3. Если вводится FPS **ниже 30** для SLIPPER:
   - скорректировать `clamp` для `Engine.max_fps` так, чтобы **только SLIPPER** мог опускаться на согласованный минимум (например 20), остальные режимы — без изменения прежних ограничений;
   - рассмотреть согласование `physics_ticks_per_second` с FPS (или оставить 30 с пометкой «осознанно») — **не ломать** быстрые снаряды без проверки.
4. Если увеличивается интервал Y-sort **выше 8**:
   - расширить `@export_range` в `YSortManager.gd` до безопасного максимума (например 16 или 32);
   - в `apply_from_save_manager` обновить `clampi(ysort_every, …)` так, чтобы новый максимум для SLIPPER достижим.
5. **Не менять** поведение `MINIMAL`, `MEDIUM`, `MAXIMUM`, `CUSTOM` без необходимости; если нужна общая правка клампа — разделить ветки так, чтобы не- SLIPPER остались прежними.

## Не делать в этой задаче

- Троттлинг сохранений (TASK-012).
- Отсечение декора и сцен (TASK-013).
- Правки HUD/боя (TASK-015, TASK-016).

## Критерии приёмки

- В редакторе: при выборе «На тапке» применяются новые числа; при «Минимальный» — как до задачи (если не оговорено иное в журнале).
- Нет ошибок при старте сцены и при смене пресета из меню.
- В `optimization-aurora-wave2/JOURNAL.md` есть запись с перечислением финальных чисел для SLIPPER.
