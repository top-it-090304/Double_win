# TASK-012 — SLIPPER: троттлинг записи сохранения на диск

## Цель

Уменьшить частоту **фактических** записей файла сохранения на диск **только в режиме «На тапке»**, чтобы снизить нагрузку на CPU/IO на слабом железе, **не теряя** прогресс при критичных событиях и при выходе из игры.

## Контекст

- `honor-of-aurora/autoload/Save_manager.gd`:
  - `save_game()` — синхронная запись JSON в `user://game_save_file.save`.
  - `request_save_game_deferred()` — коалесцирует несколько вызовов **в один кадр** (`call_deferred("_flush_deferred_save_game")`), но **не** ограничивает частоту между кадрами/секундами.
- При частых событиях (движение, лут, смена состояния) теоретически возможны **много записей подряд**, если по коду вызывается `save_game()` напрямую, а не только deferred.

## Требования (MUST)

1. Ввести **минимальный интервал времени** между успешными `save_game()` **только для SLIPPER**, когда сохранение инициировано не «критичным» путём.
2. Реализовать централизованно в `Save_manager.gd` (или в малом helper в том же autoload), с проверкой `PerformancePreset.is_slipper_mode(self)`.
3. **Критичные сохранения** должны **обходить** троттлинг или **форсировать** немедленную запись. Примеры категорий (уточнить по коду при реализации):
   - выход в меню / закрытие приложения (если есть хук);
   - смерть героя, телепорт с острова, завершение квеста — если без сохранения возможна потеря прогресса;
   - любой путь, где игрок ожидает «уже сохранено» до выключения игры.
4. Для отложенных вызовов (`request_save_game_deferred`): если в SLIPPER сработал троттлинг, **не терять** намерение сохранить: отложить до **следующего допустимого окна** (таймер `SceneTreeTimer` или накопление флага) либо слива в одну запись при стабилизации.
5. Добавить комментарии: почему выбран интервал (например 2–5 секунд для «мягких» событий — подобрать эмпирически, консервативно).

## Не делать

- Не отключать сохранение полностью.
- Не менять формат `SAVE_DATA` и список полей.
- Не уменьшать частоту сохранений для **не-SLIPPER** пресетов (если только общий рефактор не сохраняет идентичное поведение для них).

## Критерии приёмки

- На «На тапке» при активном геймплее количество записей на диск за интервал **меньше**, чем до изменений (можно замерить логами за `#if DEBUG` или счётчиком).
- После покупки/важного события и немедленного Alt+F4 (или аналога) — по сценарию теста прогресс не хуже, чем ожидается по дизайну (журнал фиксирует ограничения).
- Запись в `optimization-aurora-wave2/JOURNAL.md` с описанием политики и интервалов.
