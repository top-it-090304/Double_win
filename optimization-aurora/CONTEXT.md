# Контекст: оптимизация Honor of Aurora под ОС Аврора

## 1. Что это за проект

- **Репозиторий:** `Double_win`, игра в подкаталоге **`honor-of-aurora/`**.
- **Движок:** Godot **4.4**, рендер **GL Compatibility** (`project.godot` → `config/features`, `renderer/rendering_method`).
- **Целевая платформа:** ОС **Аврора** на **очень слабом железе** (низкая CPU/GPU производительность, ограниченная память и батарея).

## 2. Бизнес-цель

Ввести дополнительный режим производительности с названием **«На тапке»** (юмористическое название «минимального» профиля). При его активации должны включаться **все** запланированные упрощения рендера, симуляции и побочных систем, согласованные в задачах из `optimization-aurora/tasks/`.

Режим не должен ломать сохранения: значение `performance_mode` в `SaveManager` уже сериализуется; новый пресет нужно добавить так, чтобы старые сохранения мигрировали предсказуемо (см. существующую логику `PerformancePreset`, `clamp_mode`).

## 3. Уже существующая архитектура производительности (важно не дублировать)

Прочитай перед правками:

- **`honor-of-aurora/autoload/performance_preset.gd`** — класс `PerformancePreset`, enum `Mode`: сейчас `MINIMAL`, `MEDIUM`, `MAXIMUM`, `CUSTOM`. Пресеты задают `Engine.max_fps`, `Engine.physics_ticks_per_second`, `YSortManager.refresh_every_frames`, VSync через `DisplayServer`.
- **`honor-of-aurora/autoload/Save_manager.gd`** — поля `max_fps`, `performance_mode`, загрузка/сохранение, вызов `PerformancePreset.apply_from_save_manager`.
- **`honor-of-aurora/ui/menu/settings_overlay.gd`** — пункты «Минимальный / Средний / Максимальный» в `OptionButton`, описания в `_update_performance_desc`.
- **`honor-of-aurora/autoload/YSortManager.gd`** — глобальная Y-сортировка группы `y_sortable`, параметр `refresh_every_frames`.
- **`honor-of-aurora/autoload/rain_system.gd`** + **`environment/weather/`** — оверлей дождя на островах при определённых условиях.

«На тапке» логически **строже**, чем текущий «Минимальный», или заменяет/дополняет его в меню — конкретное решение фиксируется в задаче **TASK-001** (единая точка правды по enum и UI).

## 4. Ограничения и принципы

- Менять только то, что нужно для оптимизации и для режима «На тапке»; не рефакторить без причины.
- Любая оптимизация, зависящая от пресета, должна читать **`SaveManager.performance_mode`** или helper на базе **`PerformancePreset`** (избегать разрозненных магических чисел).
- После изменений — кратко задокументировать в **`optimization-aurora/JOURNAL.md`** (формат записи — в шапке журнала).

## 5. Где лежат артефакты процесса

| Путь | Содержимое |
|------|------------|
| `optimization-aurora/TASKS_INDEX.md` | Статусы задач |
| `optimization-aurora/tasks/TASK-*.md` | Детальное ТЗ на задачу |
| `optimization-aurora/JOURNAL.md` | Журнал выполненных работ |

## 6. Критерий готовности «волны» оптимизации

Игрок выбирает **«На тапке»** в настройках → сохраняется → после перезапуска режим активен → измеримо снижается нагрузка (конкретные метрики — в отдельных задачах: FPS, время кадра, отключённые эффекты).
